El problema

La operación vive repartida en planillas sueltas

Cada corredora termina reconstruyendo a mano lo que el sistema debería saber solo.

🗂️

Datos dispersos

Asegurados en una planilla, pólizas en otra, comisiones de cada aseguradora en archivos sueltos.

🔁

Trabajo duplicado

La misma información se transcribe varias veces entre planillas, con errores que se acumulan.

📉

Sin vista consolidada

Reconstruir el estado real del negocio al cierre de mes es un ejercicio manual, no una consulta.

La solución

Un módulo por cada parte del negocio, todos conectados

Nada vive aislado: una póliza sabe quién es su asegurado, qué comisión generó y qué cobros tiene pendientes.

👤

Debajo del capó

Arquitectura simple, a propósito

Un monolito Flask sin capas de abstracción innecesarias — fácil de razonar, fácil de mantener, fácil de auditar.

  • Flask (Python) + MySQL / TiDB Cloud, sin ORM de por medio
  • Jinja2 + Bootstrap 5 para una interfaz consistente en toda la app
  • Roles diferenciados: Administración, Gerencia, Operaciones, Ventas
  • Catálogos administrables en vez de datos hardcodeados en el código
En producción

Desplegado en Render.com

Con soporte Docker como alternativa portable — mismo código, cualquier proveedor con contenedores. Sin dependencia de una nube específica.
